# 数据库迁移操作手册

## 迁移文件编写规范

### 文件命名
- 格式：`YYYYMMDDHHMMSS_description.sql`（如 `20260611143000_add_payment_status.sql`）
- 描述使用小写英文，简洁明了
- 每个迁移文件只做一件事

### DDL 迁移（表结构变更）
- 新增列：必须指定 DEFAULT 值或允许 NULL
- 修改列类型：确保现有数据可以安全转换
- 删除列：先标记为废弃（deprecated），至少一个版本周期后再删除
- 新增索引：使用 `CONCURRENTLY`（PostgreSQL）或 `ONLINE`（MySQL）避免锁表
- 删除索引：确认无查询依赖后再操作

### DML 迁移（数据变更）
- 大批量数据更新必须分批执行（每批 1000-5000 行）
- 使用事务包裹，失败时自动回滚
- 记录受影响的行数，写入迁移日志

## 回滚策略

### 可逆迁移
- 每个 UP 迁移必须编写对应的 DOWN 迁移
- DOWN 迁移在测试环境验证后才能标记为可回滚
- 回滚脚本不允许包含数据丢失操作（如 DROP TABLE）

### 不可逆迁移标记
- 无法安全回滚的迁移（如删除列、合并数据）必须在文件头注释标记
- 标记格式：`-- IRREVERSIBLE: 原因描述`
- 不可逆迁移上线前需经过 DBA 审批

### 紧急回滚流程
1. 评估回滚影响范围（哪些数据会丢失或异常）
2. 执行 DOWN 迁移或恢复备份
3. 验证数据一致性
4. 通知相关团队回滚完成
5. 记录回滚原因和时间

## 上线前检查清单

### 结构变更
- [ ] 迁移文件已在测试环境执行并验证
- [ ] DOWN 迁移已编写并在测试环境验证
- [ ] 新增列有默认值或允许 NULL
- [ ] 索引创建使用在线模式（不锁表）
- [ ] 外键约束已考虑性能影响

### 数据变更
- [ ] 大批量更新已拆分为分批执行
- [ ] 更新条件已用 SELECT 验证受影响范围
- [ ] 事务隔离级别合适（避免长事务）
- [ ] 预估执行时间，超长迁移有分步计划

### 影响评估
- [ ] 评估迁移对线上服务的影响（锁表时间、CPU 负载）
- [ ] 确认迁移执行时间窗口（低峰期）
- [ ] 通知相关团队迁移计划
- [ ] 准备回滚方案和应急预案
- [ ] 监控告警已配置

## 支付相关迁移特别注意

### 金额字段变更
- 金额字段变更必须保证精度不丢失
- 从元转为分：先新增分为单位的列 → 数据迁移 → 验证 → 删除旧列
- 金额精度变更（如 2 位小数改为 4 位）必须分步执行

### 状态字段变更
- 支付状态枚举变更必须向后兼容
- 新增状态值：追加，不修改已有枚举值
- 删除状态值：先确认无数据使用该状态
- 状态机变更需同步更新业务逻辑和测试

### 交易表变更
- 交易表数据量大，结构变更需特别谨慎
- 避免全表扫描的迁移操作
- 新增索引需评估写入性能影响
- 分区表变更需提前规划
