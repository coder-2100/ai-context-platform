# 支付基础知识

## 业务术语

| 术语 | 英文 | 说明 |
|------|------|------|
| 支付单 | Payment Order | 用户发起的支付请求记录 |
| 支付流水 | Payment Transaction | 一次支付尝试的记录，一个支付单可有多次流水 |
| 收单 | Acquiring | 商户通过支付渠道收取用户资金 |
| 退款 | Refund | 将已支付资金退回用户账户 |
| 撤销 | Void | 在资金结算前取消交易 |
| 对账 | Reconciliation | 核对支付系统与渠道的账务是否一致 |
| 清算 | Settlement | 渠道将资金结算到商户账户 |
| 幂等键 | Idempotency Key | 用于保证同一请求不会被重复处理的唯一标识 |

## 核心流程

### 支付流程
1. 用户下单 → 创建支付单（状态：待支付）
2. 选择支付方式 → 创建支付流水（状态：处理中）
3. 调用支付渠道 → 等待渠道响应
4. 接收渠道回调 → 更新支付流水状态
5. 支付成功 → 更新支付单状态（已支付）
6. 通知业务系统 → 触发后续流程（发货、开通等）

### 退款流程
1. 用户/商户发起退款 → 创建退款单（状态：待退款）
2. 校验退款金额（不超过原支付金额减去已退金额）
3. 调用渠道退款接口 → 等待渠道响应
4. 接收退款回调 → 更新退款单状态
5. 退款成功 → 更新支付单可退余额

## 状态机

### 支付单状态
```
待支付 → 已支付 → 已关闭
                ↓
            部分退款 → 全额退款
```

### 支付流水状态
```
处理中 → 成功
       → 失败
       → 关闭（超时关闭）
```

### 退款单状态
```
待退款 → 退款中 → 退款成功
               → 退款失败
```

## 关键设计原则
- 支付单与支付流水分离：一个支付单可对应多次支付尝试
- 所有状态变更必须记录操作日志
- 金额使用整数（分为单位），避免浮点精度问题
- 外部交互（渠道调用）与内部状态更新必须保证最终一致性
